  LEGENDA DE GRAVIDADE:
  CRÍTICA  — Exploração trivial, impacto total.
  ALTA     — Exploração provável, impacto significativo.
  MÉDIA    — Exploração possível, reduz postura de segurança.
  BAIXA    — Má prática, impacto limitado isoladamente.

────────────────────────────────────────────────────────────────────────────────
#01 | SECRET_KEY JWT HARDCODED (INALTERADO — Ex-#01)
────────────────────────────────────────────────────────────────────────────────
Arquivo/Linha: app/users/security.py, Linha 11
Descrição:     SECRET_KEY usa fallback hardcoded
               ("a_very_secret_key_change_me_in_production"). Diferente
               das senhas admin/gatekeeper, NÃO há variável SECRET_KEY no
               .env. Isso significa que o JWT está sendo assinado com a
               chave pública hardcoded no código-fonte.
Gravidade:     CRÍTICA
Justificativa: Permite forjar tokens JWT para qualquer usuário.
               OWASP A02:2021 (Cryptographic Failures), CWE-798.
Recomendação:  Adicionar SECRET_KEY ao .env com valor gerado via:
               python -c "import secrets; print(secrets.token_urlsafe(64))"

────────────────────────────────────────────────────────────────────────────────
#02 | GATEKEEPER — COOKIE BURLÁVEL (INALTERADO — Ex-#06)
────────────────────────────────────────────────────────────────────────────────
Arquivo/Linha: app/main.py, Linhas 44-47
Descrição:     A verificação compara cookie "gatekeeper_access" com string
               "granted". Qualquer pessoa pode definir manualmente este
               cookie no navegador sem conhecer a senha.
Gravidade:     CRÍTICA
Justificativa: Elimina a camada de proteção de borda (doc/01_README.md
               §Camadas de Segurança). OWASP A01:2021, CWE-807.
Recomendação:  Usar token HMAC assinado com SECRET_KEY + timestamp.

#03 | AUSÊNCIA DE PROTEÇÃO CSRF (INALTERADO — Ex-#02)
────────────────────────────────────────────────────────────────────────────────
Arquivo/Linha: Todos os formulários POST (main.py, templates HTML)
Descrição:     Nenhum formulário contém token CSRF. Autenticação via cookies

Gravidade:     ALTA

Recomendação:  Implementar middleware CSRF ou tokens por formulário.

#25 | NOVA OBSERVAÇÃO — --forwarded-allow-ips='*' PERMISSIVO
────────────────────────────────────────────────────────────────────────────────
Arquivo/Linha: Dockerfile, Linha 23
Descrição:     O CMD de produção usa --forwarded-allow-ips='*', que confia
               em headers X-Forwarded-For de qualquer origem. Isso permite
               que um atacante forje seu IP enviando headers manipulados,
               potencialmente burlando o rate limiter baseado em IP
               (is_rate_limited) e afetando logs.
Gravidade:     MÉDIA (NOVA — introduzida pela atualização)
Justificativa: CWE-346 (Origin Validation Error). O --forwarded-allow-ips

================================================================================
  SEÇÃO 5 — TOP 5 RECOMENDAÇÕES PRIORITÁRIAS (ATUALIZADO)
================================================================================

  1. ⚠️ URGENTE — ADICIONAR SECRET_KEY AO .env
     A SECRET_KEY do JWT está usando o valor hardcoded. Todas as outras
     senhas foram fortalecidas, mas esta foi esquecida. Adicionar ao .env:
     SECRET_KEY=<valor gerado com: python -c "import secrets; print(secrets.token_urlsafe(64))">

  2. CORRIGIR GATEKEEPER — Substituir comparação de cookie por valor fixo
     por token HMAC assinado com SECRET_KEY. Sem isso, a camada Gatekeeper
     é decorativa.

  3. IMPLEMENTAR PROTEÇÃO CSRF — Middleware ou tokens por formulário.
     Priorizar: /upload, /directives POST, /users/ POST, /master-directives.

  4. FORÇAR EXTENSÃO .pdf NO UPLOAD — Ignorar a extensão do usuário e
     usar sempre ".pdf" ao salvar. A validação de magic bytes (M2) é boa
     mas insuficiente sem controle da extensão.

  5. PROTEGER EXPORTAÇÃO XLSX — Adicionar Depends(get_current_user) ou
     pelo menos check_gatekeeper ao endpoint /export/xlsx.
